## Why does `__init__.py` exist?

### 1. Package initialization

When a package is imported, Python **runs `__init__.py` first**.

```python
import src.components
```

Python internally does:

```python
execute src/components/__init__.py
```

This allows you to:

* Initialize variables
* Import submodules
* Control package behavior

### 2. Makes imports cleaner (VERY important in ML projects)

Instead of this

```python
from src.components.data_ingestion import DataIngestion
from src.components.data_transformation import DataTransformation
```

You can expose them in `__init__.py`:

```python
# src/components/__init__.py
from .data_ingestion import DataIngestion
from .data_transformation import DataTransformation
```

Now your import becomes clean:

```python
from src.components import DataIngestion, DataTransformation
```

This is **professional-grade packaging** and widely used in production ML codebases.

### 3. Controls what gets imported (`__all__`)

You can explicitly define what your package exposes.

```python
# src/components/__init__.py
__all__ = [
    "DataIngestion",
    "DataTransformation"
]
```

This prevents accidental imports and improves readability.

### 4. Logical grouping of code

In large ML projects, `__init__.py` acts like a **public API** for the folder.

## `__init__.py` in an End-to-End ML Project

Typical usage:

### Example: `src/components/__init__.py`

```python
from .data_ingestion import DataIngestion
from .data_validation import DataValidation
from .data_transformation import DataTransformation
from .model_trainer import ModelTrainer
```

Now your pipeline file becomes clean:

```python
from src.components import (
    DataIngestion,
    DataValidation,
    DataTransformation,
    ModelTrainer
)
```

This improves:

* Readability
* Maintainability
* Scalability

## Python 3.3+ Note (Important)

Python 3.3 introduced **namespace packages**, so technically `__init__.py` is not always required.

BUT
**In real-world ML & production systems**:

* Virtual environments
* Packaging tools
* Docker
* CI/CD pipelines

`__init__.py` is still **best practice**
